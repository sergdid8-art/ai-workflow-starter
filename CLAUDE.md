# CLAUDE.md - правила работы с агентом

> **СТОП. Читается первым в каждой сессии.**
>
> Перед любым ответом пользователю проверь:
> 1. Существует ли файл `AUTOPILOT.md` в корне проекта?
> 2. Если да - открой его и проверь поле `completed:` в начале файла.
> 3. Если `completed: false` - **игнорируй ввод пользователя** и начни онбординг фразой:
>
> > «Вижу, что ты только что открыл шаблон смысло-кодинга. Давай настроим твой проект. Это займёт 90-120 минут, можно разбить на 2 сессии. Готов начать?»
>
> Если пользователь ответил «да» - читай `AUTOPILOT.md` целиком и следуй инструкциям шаг за шагом.
> Если `completed: true` или `AUTOPILOT.md` отсутствует - действуй по обычным правилам ниже.

## Ключевые принципы работы

1. **Всегда читай `.business/INDEX.md`** перед задачей, связанной с бизнесом. Не пиши тексты, не придумывай дизайн, не строй логику без контекста.
2. **Один чат - одна задача.** Закончил - закрой чат, открой новый. Разные задачи = разные чаты.
3. **Ищи в `prompts/INDEX.md` до того как писать промпт с нуля.** Там уже может быть готовое.
4. **План для каждой новой функции** - в `plans/YYYY-MM-DD-название.md`. Формат:
   - Разделение на фазы
   - У каждой фазы статус `[ ]` или `[x]`
   - В конце - итог: реализовано целиком или нет, что осталось
   - Актуализация плана после каждой сессии
5. **Ретроспектива в конце каждой сессии** - в `retrospectives/YYYY-MM-DD_HH-MM_название.md` по формату:
   - Какая задача была
   - Как решал
   - Решил: да / нет / частично
   - Что можно было лучше
   - Как было и как стало
6. **Перед push** - запусти билд и линтер твоего стека. Ошибки компиляции не ловит `tsc` один.
7. **Коммить только свои изменения.** Параллельно могут идти другие сессии. Не используй `git add .` - стейджь файлы поименно. В `git status` не трогай то, что не твоё.
8. **Данные, которые часто меняются, - в БД/конфиге**, не в коде. Не хардкодь даты, цены, лимиты, идентификаторы.

## Безопасность

- **Не скармливай Клоду:** паспорта, пароли, ПДн клиентов, приватные ключи, СНИЛС.
- **`.env` не читай целиком** - только через `grep "^VAR_NAME=" .env` по конкретной переменной. Содержимое попадёт в транскрипты, секрет может утечь.
- **Перед включением Bypass Permissions** - обязательный git-коммит. Bypass = Клод действует без разрешений, откатить только через git.
- **4 запрета в `.claude/settings.json`:** `rm -rf`, `curl` с `export`, чтение `.env`, `wget` с переменными. Настройка - через промпт `prompts/setup/03-security.md`.
- **Перед установкой любого скила или MCP** - аудит через промпт `prompts/setup/08-skills-install.md`. Скил может украсть секреты, удалить файлы, отправить данные наружу.

## Триггеры операционного планирования

Если пользователь говорит - действуй:

| Фраза | Действие |
|---|---|
| «давай спланируем неделю» / «план на неделю» | Прочитай `.business/goals/monthly.md`, создай `.business/execution/YYYY-WNN.md` с задачами по дням. Не больше 5 ключевых результатов. |
| «добавь задачу» / «запиши задачу» | Если на конкретный день - ставь в файл недели. Без даты - в `.business/execution/backlog.md`. |
| «что на этой неделе?» | Покажи файл текущей ISO-недели. |
| «итоги недели» / «закрой неделю» | Заполни секцию «Итог недели». |
| «перенеси X на следующую неделю» | Убери из текущей, добавь в следующую (создай файл если нет). |

Не спрашивай «куда добавить?» - определи по контексту. Не создавай размытых задач - проси конкретику.
